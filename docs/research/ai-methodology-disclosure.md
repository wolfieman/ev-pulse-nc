## 1. Disclosure Statement

I used AI-assisted tools throughout the analytical, methodology-review, and manuscript-drafting phases of this study. AI tools accelerated specific subtasks — code generation against verified data sources, structured role-play methodology review across multiple specialized lenses, source filtering at scale during the literature review, and prose refinement of author-drafted text — but the synthesis, integration, interpretive judgment, and verification chain throughout the work are the author's. I do not appear as a co-author with any AI tool and do not credit AI for the methodological choices, the scope decisions, the interpretive framings, or the authorial responsibility for what the manuscript claims.

The AI-assisted methodology used in this study is part of a broader, deliberately systematized AI-workflow infrastructure documented in a companion repository (https://github.com/wolfieman/spring-26-mba-orchestrator), which contains a categorized AI prompt library, hub-file templates that are applied consistently across academic project repositories, workflow-automation scripts, and a documented weekly transition protocol. This study's AI usage is therefore methodology-as-infrastructure rather than ad-hoc tool use: the prompts, the role-play patterns, the verification chains, and the orchestration norms are all reusable artifacts that pre-existed this project and continue beyond it. Disclosing AI usage in this study is, in effect, disclosing the EV Pulse NC application of a recurring author-orchestrated workflow.

This disclosure aligns with the Committee on Publication Ethics (COPE) position that AI tools cannot meet authorship criteria and that AI use should be disclosed in publications, and with the practice of disclosure-without-attribution that *Nature*, *Science*, and the journals targeted for the post-semester polish of this study (e.g., *Transportation Research Part D*, *Energy Policy*) have adopted since 2023. The honest framing of the limitations of the AI-assisted methodology used in this study — that the data quality panel and the manuscript advisory panel processes constitute *internal methodology review*, not *external peer review* — is stated in Section 3 below and carried forward from the federal-compliance review of this manuscript.

## 2. AI Tool Stack

This study used the following AI tools, each in a documented role.

**Claude Code (Anthropic, CLI).** Primary technical and analytical assistant. Used for: code generation and refactoring of the Python analysis scripts (`code/python/analysis/`), data quality review (the five-expert role-play methodology documented in Section 3 below and at the dispatch-template artifact noted in Section 5), manuscript-preparation synthesis (source-read assessments, to-lift checklist maintenance, outline drafting, section drafting), figure-generation code (fig-43 through fig-45), and verification of analytical outputs against source data via the Claims-to-Evidence Matrix that anchors the data quality review. Claude Code is the most integrated AI tool in the analytical pipeline — its outputs (committed code, committed scripts, committed CSVs, committed figure PNGs/PDFs) are persisted as repository artifacts and are reproducible independent of the AI tool that generated them.

**Claude.ai (Anthropic, web app, "Claude Browser").** Secondary AI assistant used as a writing-polishing layer and as a parallel-review source for methodology and framing decisions. The Claude Browser sessions provided independent second-opinion review of key choices (the Theil novelty narrowing, the two-tier architecture framing, the panel-decision reconciliation) and surface-level prose refinement of author-drafted text. Outputs from these sessions are not directly committed to the repository; their influence is mediated through author-revised manuscript prose.

**ChatGPT (OpenAI, web app).** Broad orchestration and advisory layer. Used for high-level project framing, decision support across competing methodological choices, and structured workflow capture across the broader research process. ChatGPT's role in this specific study was supplementary to Claude Code's role; for this manuscript's analytical and drafting work, Claude Code carried the heavier load.

**Perplexity AI.** Deep research and citation-finding. Used for the literature review (Manuscript Section 3), particularly for the prior-work check on the Theil novelty claim that surfaced Choi, Xu, and Jiao (2025) and for source discovery on adjacent transportation-equity work that the literature review draws on. Perplexity's citation-oriented search results were filtered by the author and the most relevant sources were then processed through NotebookLM (below) for synthesis.

**NotebookLM (Google).** Document synthesis layer. Used for grounded synthesis of research papers, methodology documents, and primary-source materials identified through Perplexity. NotebookLM's synthesis outputs are read by the author and integrated into the literature review and methodology framing, not lifted directly into the manuscript.

**Gamma.** Used to generate the competition presentation deck that accompanies this study (`ev-pulse-nc-prioritizing-investment.pptx`). Gamma is not used in the manuscript itself; the deck and the manuscript are separate artifacts that draw on the same underlying analytical results.

**Supporting (non-AI) stack.** The analytical pipeline runs on Python (managed via `uv` with a committed dependency lockfile; primary libraries: pandas, NumPy, GeoPandas, Shapely, statsmodels, Matplotlib), VS Code as the development environment, and GitHub as the version control and reproducibility surface. The SAS Model Studio forecasts that feed Phase 1 were generated in a prior project (the Fall 2025 *nc-ev-atlas* baseline analysis); the present study consumes those forecasts as committed CSV inputs and does not run SAS as part of the current pipeline.

## 3. Workflow Integration

This section documents how AI tools fit into the research process across the five phases of the analytical pipeline plus the methodology review and manuscript drafting phases.

**Analytical phases (Phases 1 through 5).** AI assistance in the analytical phases is concentrated in code generation, code refactoring, and intermediate-result interpretation. The pattern in each phase was: I defined the analytical question and the verification target; Claude Code generated initial Python implementations against the data sources; I reviewed, refined, and re-executed the code; the outputs (committed CSVs, validation suites, and figures) were then verified independently against the source data via the Claims-to-Evidence Matrix in the data quality review (`docs/research/data-quality-review.pdf`, §8). No analytical claim in this manuscript depends on Claude Code's correctness — the claim depends on the correctness of the code I committed and the values that code produces against publicly committed raw data.

**Literature review (Manuscript Section 3).** AI assistance in the literature review took the form of a *research filter*. Perplexity surfaced candidate sources from the open web; NotebookLM synthesized clusters of related papers I had identified; I read the actual primary sources where the literature-review claims depend on them (notably for the Theil novelty check that surfaced Choi et al., 2025, and the Shorrocks and Bourguignon decomposition methodology). The AI-assisted filtering reduced the volume of reading without replacing it; the author's literacy with the underlying methodology is what the literature review claims rest on.

**Data quality methodology review (the five-expert panel).** This is the most distinctive AI-assisted methodology in this study and the one that most warrants explicit disclosure. I structured a five-perspective methodology review of the analytical pipeline using a role-play framing in which Claude Code adopted, in turn, five specialist personas: Senior Data Quality Engineer (DAMA-DMBOK), Spatial / GIS Data Specialist, Federal Data and Policy Compliance Reviewer, Econometrician and Forecast Validation Specialist, and Business / Data Analytics Strategist. Each persona reviewed the analytical pipeline against the specific roadmap sections relevant to its specialty, with verification anchored to a fixed repository commit and to a Claims-to-Evidence Matrix that pre-registered the numerical claims under review. The dispatch template that operationalized this methodology, the five resulting reports, and the cross-briefings synthesis are working artifacts of the review process; the consolidated, scrubbed deliverable that summarizes and integrates them is `docs/research/data-quality-review.pdf` (a roughly 27,000-word internal review report).

The methodology is internally rigorous in a specific sense: outputs are verifiable, the verification chain anchors each finding to a commit SHA, and the cross-briefings document attests that zero items remained open at the end of Round 1 across roughly 60 reviewed items. *This methodology is internal methodology review, not external peer review.* The manuscript's claims have not been audited by external federal program officers, by external Justice40 compliance reviewers, or by external econometricians outside this AI-assisted internal process. This distinction is deliberate and is preserved in the manuscript's Limitations section (Manuscript §8).

**Manuscript advisory panel (the four-panelist outline review).** A second AI-assisted role-play methodology was applied to the manuscript outline before drafting began. Four specialist personas — Academic Publication and MBA-to-Journal Manuscript Development; Federal Policy and Justice40 Compliance Research; Quantitative Methodology and Inequality Measurement; Publication Figure Design and Data Visualization — reviewed the proposed outline and returned structured recommendations that the author adopted, refined, or rejected on a case-by-case basis. Same methodological framing: internal review, not external peer review.

**Manuscript drafting.** AI assistance in the drafting phases of this manuscript took three forms: source-read synthesis (Claude Code reading the candidate source documents and producing assessment notes the author then approved or revised), section drafting (Claude Code drafting initial prose against section-level content commitments captured by the author, with the author then revising and editing), and prose-polishing (Claude.ai browser sessions reviewing draft prose for clarity, transitions, and tone). The final prose is author-reviewed and author-authored; AI-drafted text that survives in the manuscript does so because the author retained it after revision.

**AI in the broader workflow infrastructure.** As noted in Section 1, the AI-assisted methodology used here is part of a broader, deliberately systematized AI-workflow infrastructure visible in the companion orchestrator repository. This contextualization matters because it shows the methodology is *reusable* and *governed*, not ad-hoc: the prompts, the templates, the verification chains, and the role-play patterns are reusable artifacts that pre-existed this project and that other projects in the author's portfolio (course repos, weekly-transition protocols, optimization protocols) apply the same disciplined way.

## 4. Authorship and Attribution

Sole authorship rests with Wolfgang Sanyer. AI tools used in this study are research and writing aids, not co-authors. This position aligns with the Committee on Publication Ethics (COPE) authorship guidance, with the *Nature* and *Science* policies on AI use in scholarly publications, and with the author's deliberate policy refusing AI co-authorship while embracing AI transparency.

The author-AI boundary in this study is as follows.

| The author does | AI accelerates |
|---|---|
| Defines scope, target audience, methodological choices, scoring weights, decision thresholds | Drafts initial code implementations against author-defined data sources |
| Verifies every numerical claim against source data (Claims-to-Evidence Matrix; `docs/research/data-quality-review.pdf` §8) | Generates analytical outputs subject to author verification |
| Synthesizes across analytical phases and across the methodology review process | Provides structured multi-perspective review via the role-play methodology (Section 3) |
| Makes interpretive decisions (county archetypes, two-tier architecture framing, scope decisions) | Surfaces candidate framings the author accepts, rejects, or refines |
| Locks methodological framing (notably the Theil-decomposition-as-diagnostic-not-input boundary, Manuscript §4.4) | Proposes drafts; author approves or revises |
| Authors and signs the manuscript and accepts authorial responsibility for its claims | Does not appear as co-author and accepts no responsibility for the manuscript |

The boundary above is deliberate. The author orchestrates the AI workflow rather than being orchestrated by it. As one specific illustration of how the boundary is operationalized in the broader workflow infrastructure: the orchestrator repository's project-context document explicitly delegates a single, narrowly-scoped micro-responsibility to Claude Code (automated billing-mode management between Pro subscription and Platform usage, with author-defined switching rules and spending alerts). That is the only context in the documented workflow where an AI tool is described as "in charge" of anything, and the responsibility is bounded by explicit author-defined rules. Every other decision — scope, methodology, interpretation, authorship — sits with the author. This pattern (narrow, rule-bound AI delegations within an author-orchestrated workflow) is the reference architecture for how AI use is governed in this study.

## 5. Reproducibility Implications

The reproducibility of this study's analytical claims does not depend on access to the AI tools used during its production. All code outputs generated with AI assistance are committed to the repository (`code/python/analysis/`); a reviewer can re-execute the analysis on a clean clone using `uv` to install the committed dependency lockfile (`uv.lock`) and re-run the published scripts against the committed raw data files. The data quality review (`docs/research/data-quality-review.pdf`) anchors its findings to a specific repository commit (HEAD `be2a27b`, April 10, 2026) that is preserved in the public git history; that commit is the audit point a reviewer would re-derive against.

Manuscript-preparation working documents are local-only (gitignored) working artifacts; the conclusions they produced surface in the manuscript body and in the appendices. Reproducibility of the *analysis* therefore does not depend on AI tooling. Reproducibility of the *exact phrasing of this manuscript* would require a similar AI-assisted drafting process and is not claimed; what is claimed is that the analytical artifacts, the verification chain, and the methodology-review process are reproducible and inspectable independent of the AI tools used during production.
